## 04-01

exceljs를 사용해 보았다.  
분명 좋은 라이브러리였다. 하지만 내가 방법을 못찾은 것인지 모르지만 아쉬운 점은 column 또는 row에 데이터를 넣을 때 해당 cell의 데이터는 설정할 수 있지만 style 관련 설정을 해주지 못한다는 점이였다.

```javascript
// ref: https://www.npmjs.com/package/exceljs?source=post_page---------------------------#rows
const rows = [
  [5, "Bob", new Date()], // row by array
  { id: 6, name: "Barbara", dob: new Date() },
];
// add new rows and return them as array of row objects
const newRows = worksheet.addRows(rows);
```

위와 같이 각 row 데이터를 설정할 때 각 cell의 값은 설정할 수 있지만 merge나 style 설정은 못해주었다.
이 때문에 데이터를 가공하는 단계에서 각 객체의 특성에 맞게 style이나 merge 데이터를 객체 데이터로서 가지고 있지 못한다.

```javascript
// ref: https://www.npmjs.com/package/exceljs?source=post_page---------------------------#merged-cells
// merge by top-left, bottom-right
worksheet.mergeCells("K10", "M12");

// merge by start row, start column, end row, end column (equivalent to K10:M12)
worksheet.mergeCells(10, 11, 12, 13);
```

merge를 다음과 같이 worksheet에서 관리해주고 있다.  
따라서 데이터 가공 단계에서 각 객체가 style, merge 데이터를 갖고 있는 것이 아니라 모듈 전역 공간에서 사용하는 registry 객체에서 관리하도록 했다.  
그러면 데이터 가공을 통해 각 rows 데이터가 만들어지고 registry 객체에서는 merge나 style 정보들을 가지고 worksheet을 설정해주도록 유도했다.

## 04-06

IE와 Chrome 브라우저에서 clipboardData 차이에 대해 다루었다.  
간혹, 복사 붙여넣기의 경우 clipboardData에 해당 하는 값을 검증하거나 변환하는 작업이 필요하다.  
이런 로직을 하나의 코드 베이스로 구현하여 관리하면 좋지만 멀티 브라우저를 지원한다고 하면 쉽지 않다.  
그 이유는 다음과 같다.

1. IE에서는 window 객체에서 관리 / Chrome에서는 event 객체에서 관리
2. IE에서 clipboardData의 값을 가져올 때 Text format만 가능하다. getData('Text') / Chrome에서는 text, html 여러 format의 데이터를 가져올 수 있다. getData('text/html')
3. 만약, image 데이터가 포함되어 있는지 확인하고 싶다면 IE는 clipboardData.files에 있는 정보를 활용 / Chrome은 clipboardData.files 또는 getData('text/html')로 가져온 정보에서 img tag를 찾아서 확인

- IE, Chrome에서 붙여넣기 시 image 데이터 검증 및 변환

## 04-12

### 작업 설명

img 태그에서 data:image의 경우 이미지 데이터를 base64로 인코딩해 넣게 되는데 데이터 양이 상당하여 서버 쪽에서 저장하는데 부담이 된다.  
따라서 해당 img 태그는 공백으로 치환하는 작업을 최근에 진행했다.  
IE의 경우 clipboard에 있는 데이터를 html 태그로 가져올 수 없어서 paste 이벤트 핸들러에서 치환하는 작업에 어려움이 있었다.  
그리고 브라우저별 독립적으로 구현하는 코드를 하나의 코드 베이스로 관리하기로 했다.

### 구현

- 붙여넣기 이후에 에디터의 HTML을 가져와서 data:image의 경우 공백으로 모두 치환한 다음에 에디터에 다시 넣는다.

### 문제

- 새로운 html 정보를 innerHTML로 넣을 때 이미 존재하던 caret이 사라짐
- 사용자 입장에서는 붙여넣기 이후에 붙여넣은 데이터 뒤에 caret이 있을거라고 기대

### 해결 방법

## 04-13

### 문제

어제 진행 했던 innerHTML에 넣는 경우 caret 정보의 손실로 인해 복원하는 작업을 변경하게 되었다.  
기능 자체는 구현이 완료되었지만 예상치 못한 상황이 발생했다.  
기존 코드에서 IE일 때 예외처리 하는 코드가 있었는데 이 예외 처리로 인해 caret 정보를 복원하는 과정에서 에러가 발생했었다.  
해당 예외 처리를 분석하고 해결하는 시간이 생각보다 오래 소요될 것으로 예상되어 다른 방법을 찾게 되었다.

### 해결 방법

기존 caret 정보를 복원하는 기능을 고수할 필요가 없었기에 다음과 같이 작업했다.  
해당 Element에서 제거해야할 img Node를 찾아서 DOM에서 제거한다.(remove)  
이 경우 innerHTML로 새로운 element를 생성하여 덮어쓰지 않기 때문에 기존 caret 정보가 손실되지 않는다.

코어 자바스크립트 책을 통해 es5에서는 프로토타입을 이용하여 어떻게 상속을 흉내내었는지 배울 수 있었다.  
방법은 여러가지가 있었지만 그 중에서 더글라스 크락포드가 제안하여 널리 퍼진 방법의 원리는 다음과 같다. (편의상 생성자 함수가 아닌 슈퍼 클래스 서브 클래스로 명칭)

1. 빈 함수(Bridge) 함수의 prototype을 슈퍼 클래스의 prototype을 참조하도록 한다.
2. 서브 클래스의 prototype에 빈 함수(Bridge)의 인스턴스를 할당한다.
3. 서브 클래스의 prototype에 서브 클래스의 property를 할당한다.
4. 서브 클래스의 prototype에 constructor를 서브 클래스로 할당한다.
5. 서브 클래스의 인스턴스를 만든다.
6. 인스턴스의 **proto** - > 서브 클래스의 prototype -> 슈퍼 클래스의 prototype과 같이 된다.

## 04-14

최근 회사에서 사용하는 자체 프레임워크 교육을 준비하고 있다.
그래서 es5에서 prototype으로 클래스를 어떻게 흉내내고 사용하고 있는지, 프레임워크 단에서 event bubbling을 어떻게 구현하고 있는지 등을 공부하고 있다.

프레임워크 단에서 수행하는 event bubbling의 전체적인 흐름은 이렇다.

1. document에 프레임워크에서 관리할 이벤트 타입으로 event handler를 등록을 한다.
2. 해당 이벤트가 발생하면 event handler가 실행된다.
3. event handler에서는 event 객체를 프레임워크에 필요한 객체로 가공한다.
4. 가공할 때 DOMd에 있는 event.target 부터 상위 요소를 탐색하며 프레임워크에서 관리하는 객체들을 배열을 순차적으로 담는다.
5. 해당 배열을 순차적으로 실행하며 프레임워크에서 관리하는 하위 요소 부터 상위 요소까지 각 객체에서 정의한 event handler를 실행한다. (event bubbling)
